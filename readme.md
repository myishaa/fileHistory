# FileHistory Record Keeper

An office file record management app built with React, TanStack Router/Start, Vite, Tailwind CSS, an Express backend, and PostgreSQL.

The app is designed for tracking physical office files across divisions, with add/edit workflows, detailed search filters, dashboard reports, division settings, financial year settings, users, messages, and PostgreSQL-backed records.

## Requirements

For development:

- Node.js 20 or newer
- npm
- PostgreSQL
- `psql` command-line client

The frontend and backend are separate apps:

- Frontend: repo root
- Backend API: [backend](backend)
- Database schema: [database](database)

## Run Locally

### 1. Create PostgreSQL database

Create a database named `recordkeeper`, or use any database name you prefer:

```bash
createdb recordkeeper
```

Set the database URL in your shell:

```bash
export DATABASE_URL="postgresql://postgres:postgres@localhost:5432/recordkeeper"
```

Change the username, password, host, and database name as needed.

### 2. Run database migrations

Run every SQL file in [database](database) in number order:

```bash
psql "$DATABASE_URL" -f database/001_initial_schema.sql
psql "$DATABASE_URL" -f database/002_seed_defaults.sql
psql "$DATABASE_URL" -f database/003_auth_and_archive.sql
psql "$DATABASE_URL" -f database/004_yearly_division_allocations.sql
psql "$DATABASE_URL" -f database/005_financial_years.sql
psql "$DATABASE_URL" -f database/006_year_context.sql
psql "$DATABASE_URL" -f database/007_division_archive.sql
psql "$DATABASE_URL" -f database/008_division_merges.sql
psql "$DATABASE_URL" -f database/009_user_table_field_presets.sql
psql "$DATABASE_URL" -f database/010_table_field_preset_owner_keys.sql
psql "$DATABASE_URL" -f database/011_value_threshold_levels.sql
psql "$DATABASE_URL" -f database/012_year_selection_lock.sql
psql "$DATABASE_URL" -f database/013_indentors.sql
psql "$DATABASE_URL" -f database/014_bid_number.sql
psql "$DATABASE_URL" -f database/015_search_indexes.sql
psql "$DATABASE_URL" -f database/016_user_live_status_preferences.sql
psql "$DATABASE_URL" -f database/017_file_messages.sql
psql "$DATABASE_URL" -f database/018_message_workflow_controls.sql
psql "$DATABASE_URL" -f database/019_mmg_live_settings.sql
psql "$DATABASE_URL" -f database/020_drop_file_type.sql
```

The seed creates this initial admin login only if it does not already exist:

- Username: `ovais`
- Password: `ovais123`

Change this password immediately after the first real login.

### 3. Start the backend

Create the backend environment file:

```bash
cd backend
cp .env.example .env
```

Edit `backend/.env` if your database URL or frontend URL is different:

```env
PORT=3000
DATABASE_URL=postgresql://postgres:postgres@localhost:5432/recordkeeper
FRONTEND_ORIGIN=http://localhost:5173,http://localhost:8080
NODE_ENV=development
SESSION_COOKIE_SAMESITE=lax
```

Install backend dependencies and start the API:

```bash
npm install
npm run dev
```

The backend runs at:

```text
http://localhost:3000
```

Health check:

```bash
curl http://localhost:3000/api/health
```

### 4. Start the frontend

Open a second terminal from the repo root:

```bash
npm install
npm run dev
```

The frontend runs at:

```text
http://localhost:5173
```

By default, the frontend calls the backend at `http://localhost:3000`. If your backend is somewhere else, create a root `.env` file:

```env
VITE_API_BASE_URL=http://localhost:3000
```

## Build For Production

Build the frontend:

```bash
npm run build
```

Build the backend:

```bash
cd backend
npm install
npm run build
```

Note: the build may show a Wrangler permission warning about writing logs to `~/Library/Preferences/.wrangler`. The app build still succeeds when the command exits with code `0`.

After building:

- frontend output is generated by Vite
- backend compiled files are in `backend/dist`
- run the backend with `cd backend && npm run start`

## Deployment Notes

Use PostgreSQL for this project. The current backend is written for PostgreSQL and uses PostgreSQL-specific SQL features.

Backend production environment:

```env
NODE_ENV=production
DATABASE_URL=postgresql://...
FRONTEND_ORIGIN=https://your-frontend-domain.example
SESSION_COOKIE_SAMESITE=lax
```

Use `SESSION_COOKIE_SAMESITE=none` only when the frontend and backend are deployed on different
sites and both are served over HTTPS.

Frontend production environment:

```env
VITE_API_BASE_URL=https://your-backend-domain.example
```

Production checklist:

- Change the seeded admin password before real use.
- Keep at least one active admin user; the backend blocks deleting or downgrading the last one.
- Enable scheduled PostgreSQL backups and point-in-time recovery with the database provider.
- Lock `FRONTEND_ORIGIN` to the real frontend domain, plus any deliberate staging domain.
- Take a manual database snapshot before running new migrations or importing large data.
- Periodically test restoring a backup into a staging database.

## Air-Gapped Or Offline Deployment

If the server cannot access GitHub or the public internet, do not deploy by running `git pull` or `npm install` on that server.

Prepare a release bundle on an internet-connected machine, then copy that bundle to the server.

Recommended bundle contents:

- built frontend files
- `backend/dist`
- backend production `node_modules`
- [database](database) SQL files
- deployment scripts or service files
- `.env.example` files
- Node.js runtime installer or binary, if Node.js is not already installed on the server
- PostgreSQL client tools, if `psql` is not already installed on the server

Offline server flow:

```bash
tar -xzf recordkeeper-release.tar.gz
cd recordkeeper-release
./install-offline.sh
```

The offline install script should use only files inside the release bundle. It should not run `git pull`, `npm install`, `apt install`, or any command that needs internet access.

For an air-gapped production system, also plan local PostgreSQL backups inside the same approved environment. Database backups and uploaded/document files, if any, need separate backup rules.

## Important App Logic

### Unique Code Generation

Implemented in [src/routes/add.tsx](src/routes/add.tsx).

New files auto-generate unique code in this format:

```text
YY + DivisionCode + 3-digit serial
```

Example:

```text
2617001
```

Meaning:

- `26`: financial year `2026`
- `17`: division code
- `001`: first saved file for that division and year prefix

The serial number is calculated from existing saved files with the same prefix.

### Value Handling

In Add/Edit File:

- Value has Capital/Revenue checkboxes.
- Only one can be selected at a time.
- The value input accepts numbers and decimal points only.
- S.O. value mirrors the main Capital/Revenue selection.
- S.O. value also accepts numbers and decimal points only.

Saved fields remain:

- `valueCapital`
- `valueRevenue`
- `soValueCapital`
- `soValueRevenue`

### TCEC Rules

When `TCEC = No`:

- TCEC-related fields are disabled and cleared.
- High value related fields are disabled.
- AD is forced to `No`.
- AD-related fields are disabled.
- CNC-related fields are disabled.
- Pre-TCEC, Post-TCEC, and Refloat Post-TCEC fields are disabled.

When `TCEC = No` and Mode is not `PBM`:

- IFA fields are disabled and cleared.

### Delete Password

Implemented in [src/lib/delete-password.ts](src/lib/delete-password.ts).

The deletion password is saved in Settings and is required before deleting:

- a file
- a division

Important: this protects against accidental deletion, but it is not a replacement for user permissions and database backups.

### Theme System

Supported modes:

- White theme

Supported color tints:

- Plain white / black
- Yellow tinted

Theme classes are applied in [src/routes/__root.tsx](src/routes/__root.tsx), and CSS variables are defined in [src/styles.css](src/styles.css).

## Barcode Plan

The unique code should be used as the barcode value.

Example barcode value:

```text
2617001
```

When scanned, a scanner usually types that code into an input. The app can then find the file where:

```text
file.uniqueCode === scannedCode
```

and open:

```text
/add?fileId=<matching-file-id>
```
